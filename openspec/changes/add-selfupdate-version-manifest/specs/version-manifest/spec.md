## ADDED Requirements

### Requirement: versions.json follows a fixed schema
The version manifest SHALL be a single JSON document named `versions.json` with top-level keys `agent` and `cli`. Each section MUST contain `latest` (a semver string present as a key in `releases`) and `releases` (a map from version to release metadata). Each `agent.releases` entry MUST contain `date` (ISO 8601 `YYYY-MM-DD`), `image` (fully qualified Docker image tag), `mongo` (a map with keys `"7"` and `"4"` to exact Mongo image tags), and `notes` (human-readable summary). Each `cli.releases` entry MUST contain `date` and `notes`. The `mongo` map values MUST match the exact images consumed by the install and update engines (`mongodb/mongodb-community-server:7.0-ubuntu2204` for AVX-capable hosts and `mongo:4.4` for non-AVX hosts, per the `system-detection` spec of `add-engine-detectors`). The authoritative shape:

```json
{
  "agent": {
    "latest": "1.8.3",
    "releases": {
      "1.8.3": {
        "date": "2026-06-12",
        "image": "crenein/c-network-agent-back:1.8.3",
        "mongo": {"7": "mongodb/mongodb-community-server:7.0-ubuntu2204", "4": "mongo:4.4"},
        "notes": "Fixes bugs 1&2 Telegram"
      },
      "1.8.2": {
        "date": "2026-05-28",
        "image": "crenein/c-network-agent-back:1.8.2",
        "mongo": {"7": "mongodb/mongodb-community-server:7.0-ubuntu2204", "4": "mongo:4.4"},
        "notes": "anomaly: one notification per cycle"
      }
    }
  },
  "cli": {
    "latest": "0.1.0",
    "releases": {
      "0.1.0": {
        "date": "2026-06-12",
        "notes": "Initial release"
      }
    }
  }
}
```

#### Scenario: Valid manifest parses into resolved versions
- **GIVEN** a `versions.json` matching the schema above
- **WHEN** `internal/release` parses it
- **THEN** `agent.latest` MUST resolve to `1.8.3` with image `crenein/c-network-agent-back:1.8.3`
- **AND** the Mongo image for an AVX-capable CPU MUST resolve to `mongodb/mongodb-community-server:7.0-ubuntu2204` via key `"7"`
- **AND** the Mongo image for a CPU without AVX MUST resolve to `mongo:4.4` via key `"4"`
- **AND** the release notes for `1.8.3` MUST be `Fixes bugs 1&2 Telegram`

#### Scenario: Malformed manifest is rejected, never partially applied
- **GIVEN** a `versions.json` where `agent.latest` references a version absent from `agent.releases`, or a release entry lacks `image`, or `mongo` lacks key `"7"` or `"4"`
- **WHEN** `internal/release` validates it
- **THEN** validation MUST fail with a structured error identifying the violated field
- **AND** the consumer (`update`, TUI, `status`) MUST refuse to act on the manifest
- **AND** the CLI MUST NOT fall back to pulling `crenein/c-network-agent-back:latest`

### Requirement: The CLI release workflow publishes the manifest on every release
The release workflow of `PazNicolas/crenein-agent-tui` SHALL generate `versions.json` on every tagged release and attach it as a release asset, so the latest GitHub Release always carries the current manifest. The generated manifest MUST validate against the schema before publication, and a validation failure MUST fail the release.

#### Scenario: Release publishes an updated manifest
- **GIVEN** a new CLI tag `v0.2.0` is pushed
- **WHEN** the release workflow runs
- **THEN** it MUST generate `versions.json` with `cli.latest = "0.2.0"` and a `cli.releases."0.2.0"` entry
- **AND** it MUST include the known agent releases (sourced from the repo's agent-release seed data)
- **AND** it MUST attach `versions.json` as an asset of the `v0.2.0` GitHub Release

#### Scenario: Latest release always resolves the current manifest
- **GIVEN** multiple published releases each carrying a `versions.json` asset
- **WHEN** a consumer fetches the `versions.json` asset from the `releases/latest` endpoint
- **THEN** it MUST receive the manifest generated by the most recent release without needing to know its tag

#### Scenario: Invalid generated manifest blocks the release
- **GIVEN** the manifest generation step produces JSON that violates the schema
- **WHEN** the workflow validation step runs
- **THEN** the workflow MUST fail before uploading any `versions.json` asset

### Requirement: Agent updates resolve versions and images from the manifest
`crenein-agent update` and the TUI update flow SHALL determine which agent version to install or update to from the manifest (the explicit `--version X.Y.Z` pin or `agent.latest`), use the manifest's `image` value as the exact Docker tag to pull, select the Mongo image from the manifest's `mongo` map according to the host's AVX capability, and show the manifest `notes` in the update preview. The CLI MUST NOT pull `crenein/c-network-agent-back:latest` as an update target.

#### Scenario: Update targets the manifest's latest agent version
- **GIVEN** the manifest declares `agent.latest = "1.8.4"` with `image = "crenein/c-network-agent-back:1.8.4"`
- **AND** the running agent version is `1.8.3`
- **WHEN** the user runs `crenein-agent update`
- **THEN** the update preview MUST show `1.8.3 â†’ 1.8.4` together with the `notes` of release `1.8.4`
- **AND** upon confirmation the engine MUST be instructed to pull exactly `crenein/c-network-agent-back:1.8.4`

#### Scenario: Mongo image resolved by AVX from the manifest
- **GIVEN** the target agent release's `mongo` map is `{"7": "mongodb/mongodb-community-server:7.0-ubuntu2204", "4": "mongo:4.4"}`
- **WHEN** the host CPU lacks the `avx` flag in `/proc/cpuinfo`
- **THEN** any Mongo image reference resolved for that host MUST be `mongo:4.4` (key `"4"`)
- **AND** when the CPU has AVX it MUST be `mongodb/mongodb-community-server:7.0-ubuntu2204` (key `"7"`)

#### Scenario: Pinned agent version resolved from the manifest
- **GIVEN** the manifest contains `agent.releases."1.8.2"`
- **WHEN** the user runs `crenein-agent update --version 1.8.2`
- **THEN** the target image MUST be the manifest's `image` value for `1.8.2`
- **AND** if the requested version is absent from `agent.releases`, the CLI MUST report it and exit non-zero without pulling anything

### Requirement: Running agent version is detected via /health with a Docker fallback
The CLI SHALL detect the running agent's version by calling the public root `GET /health` on the local backend â€” no `X-API-Key` required â€” probing `https://localhost:8000/health` with insecure TLS (self-signed certificate) and falling back to `http://localhost:8000/health`, and reading the `version` field (exposed by the coordinated `c-network-agent-back` change `add-agent-health-version`). When the root endpoint answers HTTP 404 (legacy backend versions, e.g. `1.8.3`, that predate that change) or the response does not include `version`, the CLI SHALL fall back to inspecting the running agent container's Docker image tag or digest. When neither source yields a version, the agent version SHALL be reported as `unknown`.

#### Scenario: Backend exposes version in /health
- **GIVEN** the backend answers `GET /health` with `{"status": "success", "message": "Connection Successful", "version": "1.8.3"}`
- **WHEN** the CLI detects the running agent version
- **THEN** it MUST report `1.8.3` sourced from `/health`

#### Scenario: Fallback to Docker image inspection
- **GIVEN** the root `GET /health` answers HTTP 404 or its response contains no `version` field (pre-`add-agent-health-version` deployment)
- **AND** the running agent container's image is `crenein/c-network-agent-back:1.8.2`
- **WHEN** the CLI detects the running agent version
- **THEN** it MUST report `1.8.2` derived from the image tag
- **AND** when the image tag is `latest`, it MUST report the image digest and treat the semantic version as `unknown`

#### Scenario: No version source available
- **GIVEN** the backend is unreachable and Docker inspection fails
- **WHEN** the CLI detects the running agent version
- **THEN** it MUST report the agent version as `unknown`
- **AND** it MUST NOT claim the agent is up to date

### Requirement: Update availability is computed against the manifest and surfaced to the user
The CLI and TUI SHALL compare the local CLI version against `cli.latest` and the detected running agent version against `agent.latest`, and notify the user when either has a newer version available. The TUI Status view (from `add-tui-dashboard`) MUST display these notifications, and `crenein-agent status --json` MUST include the comparison fields. Availability checks MUST go through the 24h version cache (`~/.crenein/version-cache.json`).

#### Scenario: Both updates available in Status
- **GIVEN** the local CLI is `0.1.0`, the manifest's `cli.latest` is `0.2.0`
- **AND** the running agent is `1.8.3`, the manifest's `agent.latest` is `1.8.4`
- **WHEN** the user opens the TUI Status view or runs `crenein-agent status`
- **THEN** the output MUST indicate that a CLI update (`0.1.0 â†’ 0.2.0`) and an agent update (`1.8.3 â†’ 1.8.4`) are available
- **AND** `crenein-agent status --json` MUST include `cli_version`, `cli_latest`, `agent_version`, `agent_latest`, and boolean `cli_update_available` and `agent_update_available` fields

#### Scenario: Everything up to date
- **GIVEN** the local CLI version equals `cli.latest` and the running agent version equals `agent.latest`
- **WHEN** the user views Status
- **THEN** no update notification MUST be shown
- **AND** the JSON output MUST report both `*_update_available` fields as `false`

#### Scenario: Unknown agent version suppresses the agent notification
- **GIVEN** the running agent version is `unknown`
- **WHEN** update availability is computed
- **THEN** the agent comparison MUST be reported as undetermined rather than "up to date" or "update available"
- **AND** the CLI comparison MUST still be computed normally

#### Scenario: Status checks respect the cache
- **GIVEN** a `~/.crenein/version-cache.json` with `fetched_at` less than 24 hours old
- **WHEN** the TUI renders the Status view repeatedly
- **THEN** it MUST use the cached manifest data
- **AND** it MUST NOT issue GitHub API requests on each render
